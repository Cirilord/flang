# Docs Foundation Spec

## Objective

Define the first dedicated documentation structure for `flang` under `docs/`, including a language summary document, a variables document, and an improvements document for planned language work.

## Scope

In scope:

- Defining a new `docs/` directory for language documentation
- Defining `docs/language.md` as the summary and entry point for language docs
- Defining `docs/variables.md` as the documentation for variable declarations
- Defining `docs/improvements.md` as the documentation for planned improvements
- Defining the initial relationship between these documents
- Defining the first documentation topics that must be covered based on the current language stage

Out of scope:

- Implementing checker behavior
- Implementing code generation behavior
- Writing documentation for features that do not exist yet beyond a planned improvements list
- Replacing the project `README.md`

## Design

- The project must introduce a dedicated `docs/` directory at the repository root.
- The first documentation structure must include:
  - `docs/language.md`
  - `docs/variables.md`
  - `docs/improvements.md`
- `docs/language.md` must act as the language documentation summary.
- `docs/language.md` must serve as the entry point for language-specific docs.
- `docs/language.md` should describe the current implemented language surface at a high level and link to more focused docs when available.
- `docs/variables.md` must document the current variable declaration syntax implemented by the language.
- `docs/variables.md` must describe:
  - `const` declarations
  - `let` declarations
  - required explicit type annotations
  - the current numeric type naming direction
  - the currently implemented example syntax
- `docs/variables.md` must use examples aligned with the current parser implementation.
- `docs/improvements.md` must document planned future language improvements that are not implemented yet.
- `docs/improvements.md` must be used to record deferred work and follow-up ideas without redefining the current implemented behavior.
- `docs/improvements.md` should include the planned implementation of `isize` and `usize` as deferred numeric type support.
- The documentation must clearly separate:
  - current implemented behavior
  - future planned improvements
- The documentation must stay aligned with approved specs and current implementation.
- The documentation foundation must be designed so future docs such as expressions, functions, types, and code generation can be added under `docs/` without reshaping the initial structure.
- The `README.md` may reference the existence of `docs/`, but the detailed language documentation should live under `docs/`.

## Examples

- Expected documentation files:
  - `docs/language.md`
  - `docs/variables.md`
  - `docs/improvements.md`

- Example topics for `docs/language.md`:
  - what `flang` is
  - the `.f` source extension
  - the current implementation stage
  - links to variables and improvements docs

- Example topics for `docs/variables.md`:
  - variable declaration syntax
  - `const` and `let`
  - explicit type annotations
  - current numeric type examples

- Example topics for `docs/improvements.md`:
  - deferred numeric types
  - future semantic checking
  - future code generation work

- Example current syntax that should appear in the docs:

```text
const positive: i32 = 1;
const zero: u32 = 0;
let decimal: f64 = 3.14;
```

- Example deferred improvement note:
  - `isize` and `usize` are planned canonical numeric types but are not implemented yet

## Acceptance Criteria

- A spec exists defining the first dedicated documentation structure under `docs/`.
- The spec defines `docs/language.md`, `docs/variables.md`, and `docs/improvements.md`.
- The spec defines `docs/language.md` as the entry point and summary for language docs.
- The spec defines `docs/variables.md` as the current variables documentation.
- The spec defines `docs/improvements.md` as the place for planned future improvements.
- The spec requires documentation to distinguish implemented behavior from planned future work.
- The spec records `isize` and `usize` as planned improvements to document in `docs/improvements.md`.
